# Feature Specification: BGS Check Export — Health Data Check & Export Tool

**Feature Branch**: `001-check-export`

**Created**: 2026-05-16

**Status**: Draft

**Input**: BGS health data check and export tool for 43 standard files (43 แฟ้มมาตรฐาน)

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ค้นหาและตรวจสอบข้อมูลสุขภาพ (Priority: P1)

เจ้าหน้าที่โรงพยาบาลเปิดระบบและค้นหาข้อมูลสุขภาพของประชาชนในเขตพื้นที่ โดยกรองด้วยช่วงวันที่รับบริการ, รหัสหน่วยบริการ, หรือประเภทบริการ แล้วดูรายการข้อมูลที่ตรงกับเงื่อนไข เพื่อตรวจสอบความครบถ้วนก่อนส่งออก

**Why this priority**: การค้นหาและกรองข้อมูลเป็นจุดเริ่มต้นของทุก workflow หากไม่มีฟีเจอร์นี้ เจ้าหน้าที่ไม่สามารถดูหรือส่งออกข้อมูลใดๆ ได้

**Independent Test**: กรอกช่วงวันที่ → กดค้นหา → เห็นรายการข้อมูลที่ตรงกับเงื่อนไข → ระบบแสดงจำนวนรายการที่พบ

**Acceptance Scenarios**:

1. **Given** เจ้าหน้าที่อยู่หน้าค้นหา, **When** กรอกช่วงวันที่และกดค้นหา, **Then** ระบบแสดงรายการข้อมูลที่ตรงกับช่วงวันที่นั้น พร้อมจำนวนรายการทั้งหมด
2. **Given** ไม่มีข้อมูลในช่วงที่ค้นหา, **When** กดค้นหา, **Then** ระบบแสดงข้อความ "ไม่พบข้อมูล" แทนรายการว่าง
3. **Given** ค้นหาโดยไม่ระบุช่วงวันเกิด, **When** กดค้นหา, **Then** ระบบแสดงผลได้ตามปกติ (ไม่บังคับกรอกวันเกิด)
4. **Given** เจ้าหน้าที่เลือกตัวกรองหลายรายการ, **When** กดค้นหา, **Then** ระบบกรองข้อมูลตามเงื่อนไขทั้งหมดที่เลือก

---

### User Story 2 - Export ข้อมูลเป็น Excel (Priority: P1)

หลังจากค้นหาและตรวจสอบข้อมูลแล้ว เจ้าหน้าที่เลือกส่งออกข้อมูลเป็นไฟล์ Excel เพื่อใช้รายงาน, ส่งต่อหน่วยงาน, หรือตรวจสอบเพิ่มเติมนอกระบบ ไฟล์ที่ได้ต้องครบถ้วนตามมาตรฐาน 43 แฟ้มที่เกี่ยวข้อง

**Why this priority**: การ Export คือผลลัพธ์หลักของระบบ เจ้าหน้าที่ต้องการไฟล์ Excel ที่ดึงข้อมูลครบ ไม่ตัดทอน

**Independent Test**: ค้นหาข้อมูล → กด Export → ดาวน์โหลดไฟล์ Excel → เปิดไฟล์และตรวจสอบว่าข้อมูลครบถ้วนตรงกับที่แสดงบนหน้าจอ

**Acceptance Scenarios**:

1. **Given** มีรายการข้อมูลจากการค้นหา, **When** กด Export Excel, **Then** ระบบสร้างไฟล์ Excel และดาวน์โหลดโดยอัตโนมัติ
2. **Given** ไฟล์ถูกดาวน์โหลด, **When** เปิดไฟล์, **Then** ข้อมูลครบทุก field ตามมาตรฐาน ไม่มีแถวหาย ไม่มีคอลัมน์ว่างที่ควรมีข้อมูล
3. **Given** มีข้อมูลจำนวนมาก (>1,000 แถว), **When** กด Export, **Then** ระบบสร้างไฟล์สำเร็จโดยไม่ timeout
4. **Given** ไม่มีข้อมูลจากการค้นหา, **When** กด Export, **Then** ระบบแสดงแจ้งเตือน "ไม่มีข้อมูลสำหรับ Export"

---

### User Story 3 - ดูรายละเอียดข้อมูลรายบุคคล (Priority: P2)

เจ้าหน้าที่คลิกเลือกรายการในตารางผลการค้นหา เพื่อดูรายละเอียดข้อมูลสุขภาพของบุคคลนั้นๆ แบบครบถ้วน ก่อนตัดสินใจรวมหรือยกเว้นจากการ Export

**Why this priority**: ช่วยให้เจ้าหน้าที่ตรวจสอบความถูกต้องของข้อมูลรายบุคคลก่อนส่งออก ลดข้อผิดพลาด

**Independent Test**: คลิกแถวข้อมูลในตาราง → หน้าต่างรายละเอียดเปิดขึ้น → แสดงข้อมูลครบทุก field ของบุคคลนั้น

**Acceptance Scenarios**:

1. **Given** มีรายการในตารางผลค้นหา, **When** คลิกแถวรายการ, **Then** ระบบแสดงรายละเอียดข้อมูลทั้งหมดของบุคคลนั้น
2. **Given** ดูรายละเอียดแล้ว, **When** ปิดหน้าต่างรายละเอียด, **Then** กลับสู่รายการค้นหาโดยไม่สูญเสียผลการค้นหาเดิม

---

### Edge Cases

- ถ้าช่วงวันที่ที่กรอกไม่ถูกต้อง (วันสิ้นสุดก่อนวันเริ่มต้น) ระบบแสดง validation error ก่อนค้นหา
- ถ้าการเชื่อมต่อฐานข้อมูลขัดข้องระหว่างค้นหา ระบบแสดงข้อความ error ที่เข้าใจง่ายและให้ลองใหม่
- ถ้าไฟล์ Excel ที่สร้างมีขนาดใหญ่มาก ระบบแสดง progress indicator ระหว่างรอ
- ถ้า session หมดอายุระหว่างใช้งาน ระบบ redirect ไปหน้า login โดยไม่สูญเสียพารามิเตอร์การค้นหา

---

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: ระบบ MUST ให้เจ้าหน้าที่กรองข้อมูลด้วยช่วงวันที่รับบริการ (วันเริ่ม–วันสิ้นสุด)
- **FR-002**: ระบบ MUST รองรับการค้นหาโดยไม่ระบุวันเกิด (birth date เป็น optional filter)
- **FR-003**: ระบบ MUST แสดงจำนวนรายการทั้งหมดที่พบจากการค้นหา
- **FR-004**: ระบบ MUST Export ข้อมูลเป็นไฟล์ Excel (.xlsx) ที่มีข้อมูลครบถ้วนทุก field ตามมาตรฐาน
- **FR-005**: ระบบ MUST รองรับ Export ข้อมูลได้อย่างน้อย 10,000 แถว โดยไม่ timeout
- **FR-006**: ระบบ MUST แสดงรายละเอียดข้อมูลรายบุคคลเมื่อคลิกเลือกรายการ
- **FR-007**: ระบบ MUST แสดง error message ที่เข้าใจได้เมื่อเกิดข้อผิดพลาด
- **FR-008**: ระบบ MUST validate ช่วงวันที่ก่อนทำการค้นหา
- **FR-009**: ระบบ MUST รองรับ pagination สำหรับรายการผลการค้นหาจำนวนมาก

### Key Entities

- **HealthRecord**: ข้อมูลสุขภาพรายบุคคล (HN, CID, ชื่อ, วันที่รับบริการ, ประเภทบริการ, รหัสหน่วยบริการ)
- **SearchFilter**: พารามิเตอร์การค้นหา (วันเริ่ม, วันสิ้นสุด, รหัสหน่วยบริการ, ประเภทบริการ, วันเกิด[optional])
- **ExportJob**: การ Export (รายการข้อมูล, สถานะ, ชื่อไฟล์, เวลาที่สร้าง)

---

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: เจ้าหน้าที่สามารถค้นหาและดูผลลัพธ์ได้ภายใน 5 วินาที สำหรับข้อมูลในช่วง 1 เดือน
- **SC-002**: ไฟล์ Excel ที่ Export มีข้อมูลครบ 100% เทียบกับรายการที่แสดงบนหน้าจอ (ไม่มีแถวหาย)
- **SC-003**: ระบบรองรับ Export ข้อมูล 10,000 แถวได้สำเร็จภายใน 60 วินาที
- **SC-004**: เจ้าหน้าที่สามารถค้นหาและ Export ข้อมูลได้สำเร็จโดยไม่ต้องระบุช่วงวันเกิด
- **SC-005**: ผู้ใช้สามารถเรียนรู้และใช้งานระบบได้โดยไม่ต้องรับการฝึกอบรม (self-explanatory UI)

---

## Assumptions

- ระบบเชื่อมต่อกับฐานข้อมูล EHP CIS ที่มีอยู่แล้ว (ไม่สร้างฐานข้อมูลใหม่)
- ผู้ใช้งานเป็นเจ้าหน้าที่โรงพยาบาลที่มีสิทธิ์เข้าถึงข้อมูล (authentication มีอยู่แล้วหรือพัฒนาแบบ simple)
- โครงสร้างข้อมูลเป็นไปตามมาตรฐาน 43 แฟ้มของกระทรวงสาธารณสุข
- รองรับการใช้งานบน Web Browser (Chrome/Edge) บนเครื่อง Windows
- ไม่รองรับ Mobile ใน v1
- ข้อมูลที่ Export เป็น CHECK file (ข้อมูลการตรวจสุขภาพ) เป็นหลัก
- รหัสหน่วยบริการ (HOSPCODE) ของโรงพยาบาล BGS เป็นค่าคงที่หรือ configurable
